## How Replikator Works

Replikator watches your Kubernetes cluster for changes to `Secrets` and `ConfigMaps` that have been opted-in for replication. When a source resource is created or updated, Replikator pushes a mirror of that data into allowed target namespaces, or pushes it automatically out to namespaces matching centralized `ClusterReplicationRule` CRDs.

When a source is deleted, or a namespace loses its permission, Replikator cleans up any mirrored artifacts it created, ensuring your environments stay in a perfect, secure state.

## Why Replikator?

Built as a spiritual successor and modern evolution of legacy reflection tools like `kubernetes-reflector`, Replikator solves the scaling and operational bottlenecks that traditional operators face in massive multi-tenant environments.

By rewriting the replication engine heavily using Go, `controller-runtime`, strict Informer Predicates, and robust Hexagonal (Domain-Driven) Architecture, Replikator achieves flat memory profiling and eliminates the K8s API "death spiral" loop completely.

## Key Features

- **Blazing Fast & Bottleneck Free:** Utilizes custom controller predicates to drop irrelevant API events (like noisy Helm state changes) before they even reach the engine's work queue.
- **Centralized CRD Policies:** Instead of manually annotating hundreds of individual secrets, use `ClusterReplicationRule` CRDs to push global wildcard/regex rules (e.g., *sync all TLS certs from the `security` namespace to any namespace labeled `env=prod`*).
- **Validating Admission Webhooks:** Regex policies are inherently dangerous. Replikator validates CRD creation to ensure your global sync rules are structurally sound and prevents accidental data overlap.
- **Native Cert-Manager Integrations:** Out-of-the-box syncing of TLS assets generated by Cert-Manager and Ingress resources.
- **CRD Status Tracking:** Instant visibility into replication engine health directly on the Custom Resource without groveling logs.
- **API Throttling/Batching:** Safely processes massive sync workloads (e.g., thousands of secrets across hundreds of namespaces) via smart flow limits.

## Annotations Configuration

Replikator utilizes clean, active verbs for its annotations, allowing precise control over what gets synced and where.

&nbsp;

### Source Annotations (Control Policy)

Add these to the `Secret` or `ConfigMap` you wish to replicate:

| Annotation | Description | Example Value |
| :--- | :--- | :--- |
| `replikator.cloudresty.io/replicate` | Allows the resource to be requested by other namespaces | `"true"` |
| `replikator.cloudresty.io/replicate-to` | A comma-separated list of namespaces that are allowed to pull this resource. | `"ns-dev, ns-stage"` |
| `replikator.cloudresty.io/auto-replicate` | Instructs the operator to automatically push the resource. | `"true"` |
| `replikator.cloudresty.io/auto-replicate-to` | A comma-separated list of namespaces to push exactly to. Supports `*`. | `"ns-dev, ns-test-*"` |

&nbsp;

### Target/Mirror Annotations (State Tracking)

These are system-level state flags applied to the *mirrored* objects by Replikator:

- `replikator.cloudresty.io/replicated-from`: Indicates the `<namespace>/<name>` source.
- `replikator.cloudresty.io/replicated-version`: Tracks the K8s object `resourceVersion`.
- `replikator.cloudresty.io/replicated-at`: RFC3339 timestamp.
- `replikator.cloudresty.io/auto-replicated`: Indicates Replikator created this automatically.

## ClusterReplicationRule (CRD)

For enterprise platform teams, relying entirely on annotations creates an unmanageable decentralized headache. You can instead declare **Global Replication Policies** using Replikator's CRD.

```yaml
apiVersion: replication.cloudresty.io/v1
kind: ClusterReplicationRule
metadata:
  name: sync-wildcard-tls
spec:
  sourceNamespace: security-system
  sourceSelector: ".*-tls-cert$"
  resourceTypes:
    - Secret
  targetNamespaces:
    selector: "dev-.*"
    excludes:
      - dev-excluded
    includes:
      - dev-primary
  targetNameTemplate: "$1-db-credentials"
```

The CRD supports:

- **`sourceSelector`**: Regex pattern to match source resource names
- **`resourceTypes`**: Array of types to sync (Secret, ConfigMap)
- **`targetNamespaces`**: Object with `selector` (regex), `includes` (exact match list), `excludes` (list to skip)
- **`targetNameTemplate`**: Optional template for renaming resources in target namespaces (uses `$1`, `$2`, etc. from regex capture groups)

### Zero-Touch Source (No Annotation Required)

Unlike annotation-based replication, ClusterReplicationRule can sync **any** resource in the source namespace without requiring the source to have Replikator annotations. This is perfect for:

- Auto-generated secrets (ECK, RabbitMQ, MongoDB operators)
- Forked Helm charts where you can't modify annotations
- Vendor operators that strip custom annotations

*Note: CRDs update their `status.lastSyncTime`, `status.mirroredCount`, and `status.syncedResources` so you have immediate visibility into the state!*

## Security & Architecture

Replikator enforces high-end security scoping out of the box:

- **In-Memory DDD Domain Checks:** The domain entity inherently validates `source.CanReflectToNamespace(...)` before initiating a K8s client payload, eliminating unexpected permission elevations across namespaces.
- **Auto-Cleanup:** If a source `Secret` changes its policy to disallow a specific target namespace, Replikator intercepts this config change and proactively deletes the dangling mirrors in those now-forbidden namespaces.

## Deployment

### YAML Manifests (Recommended)

Replikator provides two out-of-the-box deployment options depending on your cluster's scale and availability requirements.

&nbsp;

**1. High Availability / Multi-Node (Best for Production)**
Runs multiple replicas with Leader Election enabled to guarantee zero downtime during cluster upgrades or node failures.

```sh
kubectl apply -f https://raw.githubusercontent.com/cloudresty/replikator/main/deploy/yaml/ha-multi-node.yaml
```

&nbsp;

**2. Single Node (Best for Local/Dev)**
Runs a single lightweight instance without Leader Election overhead.

```sh
kubectl apply -f https://raw.githubusercontent.com/cloudresty/replikator/main/deploy/yaml/single-node.yaml
```

## Examples

### 1. Simple Push Sync (Secret)

You have a registry credential your Dev and CI environments need:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: acr-registry-pull
  namespace: sre-tools
  annotations:
    replikator.cloudresty.io/replicate: "true"
    replikator.cloudresty.io/auto-replicate: "true"
    replikator.cloudresty.io/auto-replicate-to: "ci-systems, dev-*, testing"
data:
  .dockerconfigjson: ...
type: kubernetes.io/dockerconfigjson
```

Replikator immediately detects this and mirrors exactly identical secrets into the `ci-systems` namespace and all namespaces starting with `dev-`.

&nbsp;

### 2. ConfigMap Sync with Namespace Allowlist

Sync a ConfigMap only to specific namespaces you explicitly allow:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: production
  annotations:
    replikator.cloudresty.io/replicate: "true"
    replikator.cloudresty.io/replicate-to: "staging, uat"
data:
  DATABASE_HOST: "prod-db.example.com"
  CACHE_TTL: "3600"
```

Only namespaces `staging` and `uat` can pull this ConfigMap.

&nbsp;

### 3. On-Demand Replication (No Auto-Push)

Mark a resource as available but only replicate when another namespace requests it:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: tls-cert
  namespace: security
  annotations:
    replikator.cloudresty.io/replicate: "true"
    # No auto-replicate annotation - must be explicitly requested
data:
  tls.crt: ...
  tls.key: ...
```

Other namespaces can reference this secret in their `ClusterReplicationRule` source selectors.

&nbsp;

### 4. ConfigMap for Multiple Environments

Distribute different environment configurations across matching namespaces:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: feature-flags
  namespace: config
  annotations:
    replikator.cloudresty.io/replicate: "true"
    replikator.cloudresty.io/auto-replicate: "true"
    replikator.cloudresty.io/auto-replicate-to: "env-*,platform-*"
data:
  ENABLE_DARK_MODE: "true"
  MAX_RETRIES: "3"
  LOG_LEVEL: "info"
```

Mirrors to all namespaces matching `env-*` or `platform-*` patterns (e.g., `env-dev`, `env-production`, `platform-tools`).

&nbsp;

### 5. Kubernetes TLS Secret with Auto-Cleanup

When you remove or change the replication policy, Replikator automatically cleans up orphaned mirrors:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: platform
  annotations:
    replikator.cloudresty.io/replicate: "true"
    replikator.cloudresty.io/auto-replicate: "true"
    replikator.cloudresty.io/auto-replicate-to: "services-*"
data:
  username: dXNlcm5hbWU=
  password: cGFzc3dvcmQ=
type: Opaque
```

If you later remove `services-*` from the list, Replikator deletes the mirrored secrets from those namespaces.

&nbsp;

### 6. ClusterReplicationRule for Wildcard Sync

For platform teams managing hundreds of secrets across many namespaces, use the CRD:

```yaml
apiVersion: replication.cloudresty.io/v1
kind: ClusterReplicationRule
metadata:
  name: sync-all-tls
spec:
  sourceNamespace: security
  sourceSelector: ".*-tls$"
  resourceTypes:
    - Secret
  targetNamespaces:
    selector: ".*"
    excludes:
      - security
      - kube-system
```

This syncs all secrets ending with `-tls` from the `security` namespace to every other namespace except `security` and `kube-system`.

&nbsp;

### 7. ConfigMap Sync to Specific Included Namespaces

Use `includes` for exact namespace matching:

```yaml
apiVersion: replication.cloudresty.io/v1
kind: ClusterReplicationRule
metadata:
  name: sync-configs-to-envs
spec:
  sourceNamespace: platform
  sourceSelector: ".*-config$"
  resourceTypes:
    - ConfigMap
  targetNamespaces:
    includes:
      - dev
      - staging
      - production
```

Only syncs to exactly `dev`, `staging`, and `production` namespaces.

&nbsp;

### 8. Zero-Touch Sync for ECK Auto-Generated Secrets

Replicate auto-generated Elastic secrets without requiring annotations on the source:

```yaml
apiVersion: replication.cloudresty.io/v1
kind: ClusterReplicationRule
metadata:
  name: eck-credentials-router
spec:
  sourceNamespace: elastic-system
  sourceSelector: "^(.*?)-es-elastic-user$"
  resourceTypes:
    - Secret
  targetNamespaces:
    includes:
      - dev-backend
      - staging-backend
  targetNameTemplate: "$1-db-credentials"
```

ECK creates secrets like `prod-cluster-es-elastic-user`. This CRD syncs it to `dev-backend` and `staging-backend` namespaces, renaming it to `prod-cluster-db-credentials`. No annotations needed on the ECK-generated secret!

&nbsp;

### 9. Name Transformation with Multiple Capture Groups

For complex naming patterns, use multiple capture groups:

```yaml
apiVersion: replication.cloudresty.io/v1
kind: ClusterReplicationRule
metadata:
  name: mongo-credentials-sync
spec:
  sourceNamespace: database-system
  sourceSelector: "^(.*?)-mongo-credentials$"
  resourceTypes:
    - Secret
  targetNamespaces:
    selector: ".*"
    excludes:
      - database-system
  targetNameTemplate: "$1-app-secret"
```

Source `prod-mongo-credentials` → Target `prod-app-secret` in all matching namespaces.
